## 📢 通販素材.comと楽天必勝パッケージの概要

### 1.1 通販素材.comとは（Service Background）

通販素材.comは、EC事業者向けにドロップシッピングを提供する企業です。
ドロップシッピングとは、自社で在庫を持たずに、ネットショップで商品が売れたら、メーカーや卸業者から直接顧客へ商品を発送してもらう無在庫販売（直送販売）のビジネスモデルです。
在庫リスクを抑えつつ、手軽にECサイトを始められるメリットがあります。

主に以下の3つの価値を提供しています。

* **① 売れる商品の供給（卸）**
    * 楽天ランキング常連の人気商品、**“長く売れ続ける定番商品”** を中心にラインナップ。
    * 仕入れ、撮影、マーケット理解などの負担を店舗側が持たなくて済みます。
* **② ドロップシッピング方式**
    * **在庫リスクゼロ**で販売可能。
    * 注文発生後に商品が出荷されるため、**キャッシュフローが安定**し、保管費・在庫投資・仕入れロスが不要です。
* **③ 商品素材データの一式提供**
    * 楽天商品ページ作成に必要な素材をすべて**標準化された形式**で提供します。
        * 商品画像（PC/SP）、商品説明文（PC/SP）、商品スペック、SKU情報、注意文言・オプション情報など。
    * 店舗側は**素材収集・編集のコストを排除**できます。

▶️ **通販素材.comの本質:**
「**商品 × データ**」をパッケージ化した **EC運営インフラ** です。「売れる商品」と「すぐ使える素材データ」が揃うことで、店舗運営の立ち上がりが著しく加速します。

### 1.2 既存の問題（Legacy Issues）

通販素材.comの利用店舗の売上が伸びるほど、以下の問題が顕在化していました。

| 問題のカテゴリー | 詳細な課題 |
| :--- | :--- |
| **① 作業量が売上に比例して増加する** | 在庫反映、注文取込、ページ修正、価格変更、イベント対応など、**自動化されていない運営業務**により、売上が伸びるほど**日常業務が肥大化**する。 |
| **② 販促に集中する時間が奪われる** | イベント施策の設定、画像差し替え、SEO対策、クーポン調整など、**本来「売上を伸ばすための作業」が後回しになる**。 |
| **③ 新商品登録に割ける時間が減る** | 事務作業に追われ、人気商品をいち早く扱うために重要な**新商品の登録スピードが低下**する。 |

▶️ **まとめ:**
通販素材の商品は“売れる”ものの、**運営作業が増大するほど EC事業者の時間が奪われていた**ことが既存の課題でした。

### 1.3 楽天必勝パッケージの定義（Definition）

この課題を解決するために、通販素材.comのインフラの上に構築されたのが**楽天専用SaaS「楽天必勝パッケージ」**です。

楽天必勝パッケージは、次の課題を解決します。

* **① 楽天運営に必要な作業量を「極限まで削減」**
    * 商品登録、在庫同期、注文情報の取込、イベント施策、価格更新など、**“手動運用”が必須だった領域をシステム化・自動化**します。
* **② 新規出店者でも「最短で売れる環境」を提供**
    * 素材が揃い、自動の新商品登録、APIによる自動の在庫同期、イベント準備も自動で行われるため、**経験ゼロでも楽天運営を即スタート**できます。

## **1.4 現行仕様の課題（Existing Constraints / Sales Barriers）**

現行の楽天必勝パッケージは
“自動化によって運営負荷を大幅に削減する”
というメリットがある一方で、以下のような **構造的な制約** により、導入・運用の両方で課題が生じていた。

---

## **① リニューアル等同期時に“ひな形データ”で上書きされてしまう（営業の障壁）**

会員側では
**「価格・商品名・HTML・画像を自動更新する／しない」**
を選択できる仕組みは存在するものの、…
**⚠リニューアル時、強制的にひな形データで上書きされる**

対象：
* 商品名
* HTML（PC/SP）
* サムネイル画像
* 価格
* 項目選択肢（オプション）

店舗が時間をかけて作った独自加工（SEO調整、CTR最適化、LP構成）が
**先祖がえりする** のが現行仕様だった。

---

### **→ この仕様が、営業にとって「致命的な断られる理由」になった**

#### 🧩 **営業現場で繰り返される会話（Beforeストーリー）**

**営業：**
「楽天必勝パッケージをご案内できればと思いまして…」

**会員：**
「新商品が自動追加されるのは助かるんですが…
うちは **商品名もHTMLもサムネイルも、全部こだわって作っていて**…
接続時に **自動で上書きされるのは本当に困ります。**」

**営業（心の声）：**
（またHTMLか…これが変えられない限り、何回行っても断られる……）

**会員：**
「あと短時間とはいえ、**価格が勝手に変わるのもダメです。
利益率が崩れるので。**」

**営業：**
「承知しました……改善されたらご案内します…」

---

### ▶️ **結果：自動化の利点より“独自性の毀損リスク”が恐れられる状態に**

* SEO施策が消える
* LPの構造が壊れる
* サムネイルが統一化されCTRが落ちる
* 利益率が乱れる
* 商品オプションも上書きされる（後述）

---

---

## **② 新規加盟店：導入成功率は高いが、カスタマイズ不可に対する不満**

新規加盟店は、既存データがないため
**自動商品登録 → 即運用開始**
の流れがそのまま刺さり、導入成功率は非常に高い。

しかし、その一方で…

---

### **⚠ “変更したいのに変更できない” という不満が多かった**

例：

* 画像を差し替えたい
* 文言を季節やキャンペーンに合わせて変えたい
* SEO向けにタイトルや構造を変更したい
* 価格を変更してから販売したい

しかし現行仕様では、
**自動化領域がおおく、シームレスな登録が難しい**

---

---

## **例）商品オプション（項目選択肢）が固定で、店舗独自施策が実施できない**

「商品オプション（項目選択肢）」とは、
**楽天の商品購入画面で日時指定などを選ぶための選択肢** を指す。

※日時指定、備考欄、配送希望時間帯など

---

### **現行仕様の仕組み**

* これは **APIでひな型と連携** しており、
* 運送会社変更時などには **APIで時間帯を自動書き換え** している。

結果として、

### **→ 会員の手間は大幅に減る（メリット）**

しかし一方で…

---

### **⚠ 店舗独自の施策（例えばレビューキャンペーン）が一切実施できない**

理由：

* 項目選択肢を **自由に書き換えられないため**
* 「レビューを書いてくれた方には〜」等の
  　**独自文言を追加できない**
**独自文言とは、、、

つまり、

### **➡ 自動化が強すぎて、独自マーケティングの自由度が奪われている**

---

### ▶️ 商品オプションにおける本質的課題

| 観点  | 内容                           |
| --- | ---------------------------- |
| 自動化 | 運送会社の仕様変更に即応できる（メリット）        |
| 独自性 | 店舗独自の文言・選択肢が入れられない（致命的デメリット） |
| 影響  | レビュー施策・アップセル・購入体験の最適化が制限される  |

現行仕様は以下のように“自動化の強さ”が営業障壁になっていた：

1. 初回同期で **強制上書き**（最大の拒絶理由）
2. 新規加盟店も **カスタマイズ不可** を不満に感じる
3. 商品オプションも **自動化固定で、独自施策が実施できない**

# # **2. 4階層カスタマイズ仕様（Customization Layer Model）**

本システムでは、商品データに対する上書き・カスタマイズを
**4つの階層（レイヤー）** に分けて管理する。

これにより、
**店舗全体 → 企画単位 → 商品URL単位 → 販売単位番号（SKU）単位**
という現実の EC 運用に即した柔軟な情報管理を実現する。

---

# ## **2.1 レイヤー構造（Layer Definitions）**

本システムが採用する階層は以下の通り：

| レイヤー                                  | 説明                            | 粒度    |
| ------------------------------------- | ----------------------------- | ----- |
| **① 店舗レベル（Shop Level）**               | 店舗全体で共通化したい文言・設定を管理する最上位レイヤー  | 最も抽象的 |
| **② 企画レベル（Campaign / Project Level）** | 特定の企画（例：セール・シリーズ企画）ごとに文言を調整   | 中間    |
| **③ 商品管理番号（商品URL）レベル**                | 商品URLごとに個別最適化。SEO対応や個別LP対応に使用 | 具体的   |
| **④ 販売単位番号レベル（サイズ/カラー/SKU）**          | SKUごとの価格・在庫・属性の違いを管理する最下層     | 最も具体的 |

---

# ## **2.2 レイヤー優先順位（Rule of Override Priority）**

複数レイヤーで同じパラメーターが設定されている場合は、
**より具体的なレイヤーが優先される。**
より末端のレイヤーほど、情報が具体的なので強い
＊（重要）入力しない／カスタマイズからの除外　を選択できるようにする。

優先順位（強 → 弱）：
> # **販売単位番号 > 商品URL > 企画 > 店舗**

### 例：

```
ひな型：「棚コンセント付き引き出し収納ベッド Ever X エヴァー エックス」
店舗レベル： 「【大人気】」
企画レベル： 「【KASANE】」
商品URLレベル： 「入力なし」

→ 最終表示される商品名: 「【KASANE】棚コンセント付き引き出し収納ベッド Ever X エヴァー エックス」

誤りの例：「【大人気】【KASANE】棚コンセント付き引き出し収納ベッド Ever X エヴァー エックス」
→ 上位レイヤー（店舗）の情報を混ぜている。優先順位ルールに反する。
誤りの例：「棚コンセント付き引き出し収納ベッド Ever X エヴァー エックス」
→ 企画情報が反映されていない。上書きルールを無視している。

```

これは **より正確で意図に沿ったデータを表示するための仕様** である。

---

# ## **2.3 パラメーターごとの対応レイヤー（Where Can Customize?）**

表の解説を要件定義として整理した形：

| パラメーター                      | 店舗  | 企画  | 商品URL | 販売単位 | 備考                |
| --------------------------- | --- | --- | ----- | ---- | ----------------- |
| **商品名（title）**              | 可能  | 可能  | 可能    | 対象外  |    |
| **スマートフォン説明文（sp）**          | 可能  | 可能  | 可能    | 対象外  |          |
| **PC説明文（salesDescription）** | 可能  | 可能  | 可能    | 対象外  |         |
| **商品画像**                    | 対象外 | 対象外 | 可能    | 対象外  | 画像はURL固有のため最下層で管理 |
| **商品オプション（項目選択肢）**          | 可能  | 可能  | 可能    | 対象外  | レビューキャンペーン対応で重要   |
| **販売価格（standardPrice）**     | 可能  | 可能  | 可能    | 可能   | SKUごとに価格を変えられる    |
| **属性情報（specs）**         | 対象外 | 対象外 | 対象外   | 可能   | SKU固有（サイズ素材など）    |

---

# ## **2.4 編集方式（先頭／末尾／完全独自）**

文章系パラメーターは、レイヤーに応じて編集方式が異なる。

| 編集方式               | 説明             | 主な用途          |
| ------------------ | -------------- | ------------- |
| **先頭追加（Prefix）**   | 文頭に店舗共通文言などを挿入 | 店舗レベルの統一表現    |
| **末尾追加（Suffix）**   | 文末に注意文や企画文を追加  | 企画ごとのキャンペーン表現 |
| **完全独自（Override）** | 元文を丸ごと上書き      | 商品URLの独自SEO対策 |

---

# ## **2.5 設計意図（Why 4 Layers?）**

この 4階層モデルは、以下の要件を満たすために設計されている。

---

### **① ブランド表現の統一と自由度の両立**

* 店舗単位：ブランドの基調を守る
* 商品URL単位：SEO対策に必要な個別性を保持

両方を成立させる必要がある。

---

### **② 企画施策を一括反映するため**

セール企画、ブランド企画など、**複数商品に共通する宣伝文**を一括で管理できる。




# ## **2.6 4階層反映ロジック（最重要ルール）**

要件として必ず明記すべき仕様：

> # **複数レベルに設定がある場合、より具体的なレイヤーの値を採用する。**

優先順位：
**販売単位番号 > 商品URL > 企画 > 店舗**

---

# ## **2.7 これにより実現するメリット**

### ✔ SEOが崩れない

（商品URLレベルで独自に最適化した内容が守られる）


### ✔ 企画施策を高速で展開できる

（Suffix で注意文・訴求文の一括変更）

### ✔ SKU ごとの正確な販売情報を保持できる

（価格・属性が SKU 単位で持てる）

### ✔ 上書き事故が起きない

（優先順位ルールが衝突を防ぐ）

---




# # **3章：URL登録業務フロー仕様（URL Registration Process Specification）**

本章では、URL登録に関わる一連の業務プロセスを、
**システム仕様として AI も解釈できる論理構造**で定義する。

業務フローは大きく以下の4フェーズで構成される：

> **① 初期化（Initialization）**
> **② スコープ確定（Scoping）**
> **③ カスタム設定（Customization）**
> **④ 登録・反映（Execution / Update Loop）**

---

# ## **3.1 フェーズ構造概要（Process Phase Overview）**

### **フェーズと目的**

| フェーズ         | 目的                             | 出力（次フェーズへの引き渡し） |
| ------------ | ------------------------------ | --------------- |
| **① 初期化**    | 楽天RMSと通販素材を接続し、操作可能な状態を作る      | 認証済みセッション       |
| **② スコープ確定** | 登録対象の商品群 → URL → SKU を階層的に決定する | 対象SKUの一覧（最小粒度）  |
| **③ カスタム設定** | 各階層に応じた上書きロジックで編集内容を決定する       | 最終的に反映される商品データ  |
| **④ 登録・反映**  | RMS APIで商品登録し、完了後は再編集可能な状態に戻る  | 登録完了ステータス       |

---

# ## **3.2 詳細プロセス仕様（Step-by-Step Specification）**

以下に、業務フロー①〜⑮を正式仕様として示す。

---

## ### **① 初期設定を行う（Initialization）**

**目的**

* RMS API 認証情報の登録
* 楽天店舗とのセッション確立

**入力**

* RMS APIキー
* FTP情報
* 店舗識別情報

---

## ### **② 店舗一律のカスタム設定を行う（Global Default Config）**

**目的**
店舗全体に共通するデフォルト設定を定義する。

**設定例：**

* 一律価格調整
* 商品名フォーマット（prefix / suffix）
* HTMLのベース構造
* 項目選択肢の基本定義

**意味**

> 後続の企画 / URL / SKU の設定は **この値を初期値として上書きする**。

---

## ### **③ 登録する企画を選択する（Campaign Scoping）**

**目的**
対象とする商品群の母集合（企画）を定義する。

**仕様**

* 複数企画の同時選択が可能
* 選択した企画は後続ステップで必ず候補となる

---

## ### **④ 登録する企画の商品を選択する（Campaign Item Scoping）**

**目的**
企画内の商品（企画商品群）から、実際に処理対象とする商品を絞る。

---

## ### **⑤ 登録する商品URL（商品管理番号）を指定する（URL Scoping）**

**目的**
企画内の商品から、商品URL単位にまでブレイクダウンする。

**仕様**

* デフォルト全選択
* URLレベルの個別設定が可能になる基点ステップ

---

## ### **⑥ 登録する販売単位番号（SKU）を指定する（SKU Scoping）**

**目的**
商品URLの中の 販売単位番号（SKU） を選択する。
販売単位番号（SKU）は最小粒度であり、価格や属性が異なる単位。

---

# ## **3.3 カスタム設定フェーズ（Customization Specification）**

---

## ### **⑦ 価格のカスタム設定（Price Customization）**

**適用階層**

* 店舗レベル
* 企画レベル
* URLレベル
* 販売単位番号（SKU）レベル（最優先）

**仕様**

* 店舗、企画、URLレベルでの一律％増減
* 販売単位番号（SKU）ごとの個別価格での上書き

---

## ### **⑧ 商品名のカスタム設定（Title Customization）**

**用途**
SEO / 検索順位最適化

**編集形式**

* 文頭追加（Prefix）
* 文末追加（Suffix）
* 完全独自置換（Override）

---

## ### **⑨ 商品LP（説明文・HTML）のカスタム設定（LP Customization）**

**仕様**

* PC/スマホHTML編集
* テキストの置換
* 注意文の追記
* セール表現の注入

**反映階層**

* 店舗
* 企画
* URL（最重要）

---

## ### **⑩ 商品画像のカスタム設定（Image Customization）**

**適用階層**
URLレベルのみ（対象外：店舗 / 企画 / SKU）

**仕様**

* サムネイル置換
* フリック画像の追加
* FTPアップロード連携
* 外部URL指定

---

## ### **⑪ 項目選択肢（オプション）のカスタム設定（Option Customization）**

**設定内容**

* オプション項目名の変更
* 選択肢文言の変更

# ## **3.4 プレビュー・反映フェーズ（Execution Specification）**

---

## ### **⑫ 登録内容のプレビュー（Preview）**

**目的**
全レイヤーの設定値が衝突なく合成されているか確認する。

**表示内容**

* 店舗設定
* 企画設定
* URL設定
* SKU設定
* 価格 / 商品名 / HTML / 画像 / オプション

---

## ### **⑬ RMS 商品登録APIの実行（RMS Execution）**

**仕様**
* updateItem APIなどを使用
* SKU単位まで完全反映
* 大量同時処理に対応

---

## ### **⑭ 完了画面（Completion）**
**表示内容**
* API処理結果
* エラー時のログ
* 成功した商品一覧

---

## ### **⑮ 既存URLの編集（Maintenance / Update Loop）**

**目的**
登録後も自由に調整できる。
